# Post Bridge — робочі нотатки

> Цей файл веде Claude як робочий журнал: логіка проекту, знайдені баги, прийняті рішення.
> Оновлюється при кожній суттєвій сесії роботи над кодом. Дивись також `TESTING_GUIDE.md`
> (інструкція для тестування — там детальний опис очікуваної поведінки кожної сторінки).

## Огляд логіки (коротко)

- **Instagram / Telegram / WooCommerce** — публікація автоматично через API (`lib/publishers/*`).
- **Prom.ua** — публікація НЕ через API (Prom не дає створювати товари через API). Юзер заповнює
  поля в `/posts/new`, після публікації на сторінці поста можна скачати `.yml` файл
  (`lib/generators/prom-yml.ts`) і імпортувати вручну в кабінеті Prom.
- **Horoshop Basic/Standard** — так само вручну, через `.csv` (`lib/generators/horoshop-csv.ts`).
- **Horoshop Pro** — теоретично може публікуватись через API (`lib/publishers/horoshop.ts`), але
  ця гілка ще не доробна (див. "Відомо, але поки не чіпали" нижче).

Дані для Prom/Horoshop зберігаються в `posts.prom_data` / `posts.horoshop_data` (jsonb) і
рендеряться в форму через `components/posts/PromFields.tsx` / `HoroshopFields.tsx`.

## Сесія 2026-07-05 — виправлення по /posts/new

Задача: підключення платформ працюють, але є проблеми на `/posts/new`. WooCommerce, Telegram,
Instagram навмисно не чіпали (нема підключення / вже перевірені).

### 1. Поля Horoshop не з'являлись при виборі платформи — ВИПРАВЛЕНО

**Причина:** компонент `components/posts/HoroshopFields.tsx` існував і був повністю робочий
(підтягує кастомні характеристики з `/api/horoshop/settings`), але він ніде не використовувався.
У `components/posts/PostForm.tsx` був імпортований і рендерився тільки `PromFields`, для Horoshop
блоку не було взагалі — ні імпорту, ні стейту, ні передачі даних в `createPost`.

**Що зробив:**
- `PostForm.tsx`: додав імпорт `HoroshopFields`, стейт `horoshopData`, рендер блоку при
  `targets.includes('horoshop')`, передачу `horoshop_data` в `createPost()`.
- `lib/posts/actions.ts`: `createPost()` тепер приймає `horoshop_data` і записує його в
  `posts.horoshop_data` при створенні поста (раніше поле просто ігнорувалось, тому CSV-генератор
  завжди отримував `null` і в CSV не потрапляли ні SKU, ні наявність, ні характеристики).

✅ **Оновлення (2026-07-05, пізніше того ж дня): колонку додано, розібрались до кінця.**
Valleri виконала `ALTER TABLE posts ADD COLUMN horoshop_data jsonb;` в Supabase. Я прибрала
тимчасову умову в `lib/posts/actions.ts` — тепер `horoshop_data` пишеться безумовно, як і
`prom_data`.

Але після цього виявився ДРУГИЙ, окремий і вже насправді "старий" баг, який існував в коді
задовго до цієї сесії і не мав жодного стосунку до відсутньої колонки: **скачування Horoshop
CSV падало з 500 для будь-якого поста, що таргетить Horoshop, навіть якщо в нього немає жодних
кастомних характеристик.**

Причина: `PATCH /api/horoshop/settings` зберігає `custom_fields` в `connected_platforms.credentials`
як **JSON-рядок** (`JSON.stringify(custom_fields)`), а не як "голий" масив. `GET /api/horoshop/settings`
це коректно враховує (парсить рядок назад у масив), а от `app/api/posts/[id]/download/horoshop/route.ts`
просто робив `credentials.custom_fields as string[]` — TS-каст нічого не робить у рантаймі, тому
`customFieldNames` фактично залишався рядком, і виклик `customFieldNames.map(...)` в
`lib/generators/horoshop-csv.ts` кидав `TypeError: customFieldNames.map is not a function`.

Знайшла це, тимчасово додавши `console.error` + повернення деталей помилки в JSON-відповідь
(прибрала після діагностики). Виправила: скопіювала той самий парсинг (`typeof raw === 'string'
? JSON.parse(raw) : ...`), що вже є в `GET /api/horoshop/settings`, у `download/horoshop/route.ts`.

Перевірено живими запитами через `fetch()` у консолі браузера на реальному (старому) пості з
Prom+Horoshop і на новоствореному тестовому пості з SKU `HRSH-TEST-001` — обидва тепер віддають
коректний CSV (200, є артикул, наявність, ціна тощо).

## Стара позначка (вже нерелевантна, лишаю для історії)

🔴 **Підтверджено живим тестом (2026-07-05): колонки `horoshop_data` в таблиці `posts` НЕМАЄ.**
Перевірив у Supabase Table Editor (`vallerii's Project`, продакшн) — в таблиці `posts` є
`prom_data jsonb`, але немає `horoshop_data`. Через це збереження поста з обраним Horoshop
падало з помилкою `Could not find the 'horoshop_data' column of 'posts' in the schema cache`
— і через безумовний запис ця помилка ламала створення БУДЬ-ЯКОГО поста (не тільки Horoshop),
бо ключ `horoshop_data` завжди був у payload вставки.

**Тимчасовий фікс (вже в коді, `lib/posts/actions.ts`)**: тепер `horoshop_data` додається в
insert тільки якщо він реально переданий (тобто тільки коли обрано Horoshop). Це повертає
робочий стан для Telegram/Instagram/Prom-постів (перевірено — чернетка з Telegram зберігається
без помилок). Але пости з Horoshop і далі не збережуть ці дані, поки колонки нема — це не
розв'язання, а стоп-гап, щоб не ламати решту платформ.

**Що треба зробити (потрібне підтвердження користувача — я не виконую сам, бо це прод-БД):**
```sql
ALTER TABLE posts ADD COLUMN horoshop_data jsonb;
```
Виконати в Supabase → SQL Editor. Після цього прибрати умову в `lib/posts/actions.ts`
(позначено коментарем `TODO`) і повернути `horoshop_data: formData.horoshop_data ?? null`
безумовно.

Тестовий чорновик "Тестовий пост Telegram (перевірка регресії)" видалила сама через нову
кнопку видалення (після того як фічу видалення додали — див. пункт 5 нижче). Тестові пости
для перевірки Horoshop CSV теж прибрала після перевірки.

### 2. Дублювання підсвітки в сайдбарі на /posts/new — ВИПРАВЛЕНО

**Причина:** `components/layout/Sidebar.tsx` вважав пункт активним через
`pathname.startsWith(href)`. Для `/posts/new` це спрацьовувало одночасно і для пункту
"Новий пост" (`/posts/new`), і для пункту "Публікації" (`/posts`), бо `/posts/new` починається
з `/posts`.

**Що зробив:** переписав логіку — тепер серед усіх пунктів меню шукається найдовший (найбільш
конкретний) збіг шляху, і активним стає лише він. Заодно виправив дрібну суміжну помилку:
раніше `startsWith(href)` міг би хибно спрацювати і на схожих шляхах без `/` (напр. `/posts` і
гіпотетичний `/postsomething`) — тепер порівняння йде з `href + '/'`.

### 3. Категорія маркетплейсу Prom — ВИДАЛЕНО

Поле було нефункціональним "по дизайну": `app/api/prom/groups/route.ts` завжди повертав
`categories: []` з коментарем "Prom не надає публічний endpoint для категорій маркетплейсу".
Тобто інпут пошуку завжди показував "Категорії недоступні" і ніколи нічого не давав вибрати.

Додатково: в `lib/generators/prom-yml.ts` це поле помилково писалось у YML як ДРУГИЙ тег
`<categoryId>` (перший — id групи товарів, обов'язкове поле; другий мав бути
`<portal_category_id>` за офіційною документацією Prom). Два однакові теги `<categoryId>` в
одному `<offer>` — невалідно і могло плутати імпорт. Видалення поля прибирає і цей баг.

**Що зробив:** видалив `marketplace_category_id` з `PromFields.tsx` (стейт, UI-блок, тип
`PromData`), з дефолтного стейту в `PostForm.tsx`, з генератора `prom-yml.ts`, прибрав мертвий
`categories: []` з `/api/prom/groups`.

### 4. Перевірка полів у файлах для скачування (Prom YML / Horoshop CSV)

Звірив поточну генерацію з офіційною документацією Prom та Horoshop.

**Prom YML (`lib/generators/prom-yml.ts`)** — знайшов і виправив: поле "Артикул (SKU)", яке
юзер заповнює в формі, використовувалось ТІЛЬКИ як `id` офера, і ніколи не потрапляло в тег
`<vendorCode>` (це поле, яке Prom показує як "Артикул" у картці товару і використовує для
пошуку). Додав `<vendorCode>{sku}</vendorCode>`, коли SKU заповнено.

Решта обов'язкових тегів присутні: `<name>`, `<name_ua>`, `<description>`, `<price>`,
`<currencyId>`, `<categoryId>` (з fallback `"1"`, якщо групу не вибрано — це задокументована
поведінка).

**Horoshop CSV (`lib/generators/horoshop-csv.ts`)** — колонки: Назва, Артикул, Ціна, Стара ціна,
Валюта, Наявність, Кількість, Опис, Фото + кастомні характеристики. Це відповідає задокументованій
поведінці.

⚠️ **Не виправляв, але важливо знати:** за офіційною документацією Horoshop
(help.horoshop.ua), для імпорту НОВОГО товару обов'язкові 3 поля: **SKU, Назва, Категорія**.
У нашому CSV немає колонки "Категорія" взагалі — це вже задокументовано в `TESTING_GUIDE.md`
як свідоме обмеження ("Horoshop Basic без розділів — розділ вказується вручну після імпорту") і
в планах ("Horoshop — розділи: тільки для Pro, підтягувати з API"). Тобто це не баг, а
недороблена фіча, з якою користувач вже ознайомлений. Просто фіксую тут, бо це реальний ризик:
якщо в Horoshop категорія дійсно обов'язкова для СТВОРЕННЯ нового товару (а не тільки для
оновлення), імпорт нових товарів без категорії може падати з помилкою — вартувало б перевірити
на реальному акаунті Horoshop.

Також: `lib/publishers/horoshop.ts` (авто-публікація через API для Horoshop Pro) взагалі не
використовує `horoshop_data` (артикул, стара ціна, наявність, кількість, характеристики) — там
захардкожено `article: SKU-${Date.now()}` і `presence: 'В наявності'`. Це узгоджується з
TESTING_GUIDE ("Horoshop Pro — 🔜 API публікація, ще не реалізовано"), тому не займався цим
зараз, але тут можна зекономити час наступного разу, якщо дійдуть руки допиляти Pro-публікацію.

### 5. Видалення постів — ДОДАНО

Запит: потрібна можливість видаляти пости з БД (виявили на тестовому пості, який я створила
під час перевірки). Додала:
- `deletePost(id)` в `lib/posts/actions.ts` — серверний екшн, видаляє з перевіркою
  `eq('user_id', user.id)`, щоб не можна було видалити чужий пост підміною id.
- `components/posts/DeletePostButton.tsx` — кнопка-кошик 🗑️. Підтвердження зроблено як
  **два кліки поспіль** (перший клік → кнопка на 3 секунди перетворюється на "Точно видалити?",
  другий клік підтверджує), а НЕ через `window.confirm()`. Це свідоме рішення: нативний
  `confirm()` виглядає чужорідно на фоні темного UI застосунку, і додатково він блокує
  JS-рендеринг сторінки (під час тестування через браузерну автоматизацію нативний діалог
  вішав вкладку — зайвий доказ, що краще уникати блокуючих діалогів у веб-застосунку).
- Кнопка додана в список постів (`PostList.tsx`, картка) і на сторінку поста
  (`app/posts/[id]/page.tsx`, шапка, з редіректом на `/posts` після видалення).
- Заразом виправила дрібний суміжний баг: у `PostList.tsx` в `PLATFORM_META` не було запису
  для `horoshop` — картки з таргетом Horoshop не показували бейдж платформи в списку постів
  (на сторінці окремого поста бейдж був, там `PLATFORM_META` вже містив horoshop).

### 6. Розділ каталогу Horoshop — ДОДАНО (гібридний підхід для Basic, живий API для Pro)

За результатом обговорення: логіка розділяється по тарифу, який вже й так зберігається в
`credentials.plan` при підключенні (`HoroshopModal.tsx`).

**Pro/b2b (є API):** знайшла в офіційній документації Horoshop
(https://horoshop.notion.site/API — розділ "Сторінки (категорії)" → "Експорт категорій")
функцію `POST /api/pages/export/` з параметрами `{token, parent}` — повертає дочірні розділи
відносно `parent` (за замовчуванням корінь, `parent: 0`). Це НЕ плаский список як у Prom
(`/groups/list`) — доводиться рекурсивно обходити дерево розділ за розділом. Реалізувала це в
новому `app/api/horoshop/categories/route.ts`: BFS від кореня з обмеженням (макс. 800 вузлів,
6 рівнів вкладеності — запобіжник від зависання на величезних каталогах), результат — плаский
список з `fullPath` (типу "Іграшки / Розвиваючі"), так само як `buildGroupPath` в `PromFields.tsx`.
У `HoroshopFields.tsx` для `plan === 'pro'` це відображається як живий пошук з випадаючим
списком — один в один як "Група на сайті" в Prom.

Також додала: `lib/publishers/horoshop.ts` (авто-публікація через API) тепер передає
`parent: { id: category_id }` в payload `/api/catalog/import/`, якщо розділ обрано — за
документацією Horoshop це **обов'язкове поле при створенні нового товару** через API, тому без
цього автопублікація на Pro однаково не спрацювала б коректно для нових товарів.

**Basic/Standard (нема API):** гібрид, як обговорили. У юзера є свій список розділів
(`credentials.categories`, зберігається як JSON-рядок — так само як `custom_fields`). В формі
поста (`HoroshopFields.tsx`) — пошук по цьому списку; якщо введеної назви нема — з'являється
"+ Додати новий розділ «...»", і після додавання вона одразу зберігається в `credentials.categories`
через `PATCH /api/horoshop/settings` (без сторінки, у фоні) — наступного разу вона вже в списку
для вибору. Керувати цим списком (додати/прибрати/переставити наперед) також можна в
`/platforms → ⚙️ Налаштувати характеристики товарів` — додала другу секцію "Розділи каталогу"
в `HoroshopSettingsModal.tsx`, перевикориставши компонент `HoroshopFieldsEditor.tsx` (він і так
був універсальним редактором списку, додала йому опціональні пропси для тексту/лейблів).

**Технічні зміни:**
- `lib/publishers/types.ts` — `HoroshopData` отримав `category_id?` (тільки Pro) і `category_name?`.
- `app/api/horoshop/settings/route.ts` — GET тепер віддає ще `categories` і `plan`; PATCH приймає
  часткові оновлення (`custom_fields` і/або `categories` окремо, не затираючи одне одним).
- `lib/generators/horoshop-csv.ts` — нова колонка "Розділ" (одразу після "Назва").
- `components/platforms/modals/HoroshopFieldsEditor.tsx` — став параметризованим (hint/placeholder/
  emptyLabel/saveLabel), щоб не плодити копію заради інших підписів.

**Перевірено живими тестами:** підключений акаунт в проєкті — Basic (`plan: "basic"`, підтвердила
через `GET /api/horoshop/settings`). Ввела нову назву розділу "Квіти штучні" через "+ Додати
новий розділ" у формі поста → вона одразу зʼявилась і в `credentials.categories`, і в
налаштуваннях платформи. Створила пост, скачала CSV — колонка "Розділ" містить правильне
значення. **Не чіпала** цю тестову категорію "Квіти штучні" в налаштуваннях — не видаляю дані
без прямої вказівки, а це цілком може бути реальний розділ (магазин продає штучні квіти, судячи
з інших постів) — вирішуй сама, лишити чи прибрати.

Живий Pro-пікер (`/api/horoshop/categories`) НЕ протестований на реальному Pro-акаунті — в
проєкті підключений тільки Basic. Код написаний і має правильно розбирати відповідь API за
документацією, але варто перевірити на справжньому Pro-магазині Horoshop, коли буде можливість.

### 7. Спільні поля Prom/Horoshop — прибрала дублювання

Задача: "Стара ціна", "Кількість", "Наявність", "Валюта", "Пошукові запити/теги" заповнювались
окремо для Prom і окремо для Horoshop — якщо юзер обирав обидві платформи, доводилось вводити
одне і те ж двічі.

**Що зробила:**
- Новий компонент `components/posts/SharedMarketplaceFields.tsx` — блок "🔗 Спільні поля для
  Prom.ua та Horoshop", рендериться в `PostForm.tsx` ОДИН раз, коли обрано prom і/або horoshop
  (незалежно від того, чи обрані обидві, чи тільки одна — так простіше і однаковіше).
- З `PromFields.tsx` і `HoroshopFields.tsx` прибрала дубльовані інпути (стара ціна, наявність,
  кількість, теги). В Prom лишились тільки унікальні поля: група на сайті, виробник, одиниця
  виміру, артикул. В Horoshop лишились: розділ каталогу, артикул, характеристики.
- Валюта окремо НЕ виносила — вона вже й так була спільною (одне поле "Ціна" у блоці "2. Контент"
  з вибором UAH/USD/EUR, використовується для будь-якої платформи з `needsPrice`). Просто цього
  не було видно, бо генератор Horoshop CSV раніше не мав колонки "Валюта" — та ні, мав, я
  перевірила ще раз. Реальна прогалина була не в CSV, а в живій публікації через API (Pro):
  `lib/publishers/horoshop.ts` взагалі не передавав `currency` в payload — додала.

`{ id: post.id }`. В `PostForm.tsx` доданий `useRouter()` з `next/navigation`, і після

### 4. YML писав товар "в російський варіант" — виправлено

Дослідив офіційну документацію support.prom.ua ("Імпорт через YML — формат файлу",
"Українізація: оновлений двомовний імпорт товарів"). Ключовий факт: тег `<name>` без
суфікса — це поле, яке Prom трактує як **російську** версію (історична конвенція
платформи), `<name_ua>` — окреме поле для української. Якщо в файл покласти той самий
текст в обидва теги одночасно (як робив старий генератор), Prom вважає, що юзер вже сам
надав дві мовні версії, і НЕ перекладає — тобто український текст назавжди "застряє" в
полі, яке система вважає російським.

Офіційно задокументована поведінка: якщо файл містить **лише одну** мовну версію, під час
ручного імпорту (саме так юзер і завантажує файл в Prom — вручну) Prom показує чекбокси
"якою мовою заповнений товар". Якщо відмітити "українською" для Назви/Опису/Пошукових
запитів — Prom **безкоштовно сам перекладає** товар на російську у фоні (за кілька годин).

Виправлення в `lib/generators/prom-yml.ts`: прибрав `<name_ua>`/`<description_ua>` —
тепер пишеться тільки один (український) варіант `<name>`/`<description>`/`<keywords>`.
Додав підказку в `PromFields.tsx`: нагадування відмітити чекбокси "українською" під час
імпорту в кабінет Prom (це крок, який відбувається на боці Prom і який ми не можемо
виставити за юзера програмно).

**Перевірено живим тестом:** створив пост з Прома+Horoshop, заповнив Стан/Колір/Матеріал,
посилання на категорію, кастомну характеристику "Розмір" (додану заздалегідь в
налаштуваннях Prom) — скачаний YML містить `<portal_category_url>`, три `<param>` тегі
(Колір/Матеріал/Стан) та `<param name="Розмір">`, без `name_ua`/`description_ua`. CSV
Horoshop містить нові колонки Стан/Колір/Матеріал без дублів. Прибрав тестові пости і
тестову характеристику після перевірки.

### 5. Питання про безкоштовний автопереклад для мультимовних сайтів — відповідь (без коду)

Для самого Prom.ua — переклад вже безкоштовний і вбудований (див. пункт 4 вище), нічого
додатково розробляти не треба, тільки відмічати чекбокси при імпорті.

Для мультимовних магазинів на власному сайті (Horoshop/WooCommerce з кількома мовами)
безкоштовних/бюджетних варіантів декілька, якщо це знадобиться окремою задачею:
- **DeepL API (Free)** — до 500 000 символів/місяць безкоштовно, дуже хороша якість
  перекладу, офіційно підтримує українську. Найкращий баланс якість/безкоштовний ліміт.
  Потребує реєстрації і API-ключа.
- **LibreTranslate** — повністю безкоштовний і open-source, можна самостійно
  захостити (без лімітів) або користуватись публічним інстансом (з лімітами і гіршою
  якістю за DeepL).
- **Google Cloud Translation** — має безкоштовний ліміт (з прив'язкою картки), далі платно.

Рекомендація, якщо вирішимо робити: DeepL Free API — простий у підключенні, ліміту
500k символів/міс має вистачити для обсягу постів, якість перекладу українською висока.
Це можна реалізувати окремою задачею: наприклад кнопка "Перекласти" в формі поста, яка
робить виклик до DeepL і заповнює альтернативне мовне поле.

## Сесія 2026-07-05 (продовження 4) — дублікати груп Prom при повторному імпорті

Юзер показала скріншот з кабінету Prom: пошук категорії за назвою повертав 4 однакові
"Голови георгін, хризантем" — групи дублювались після кожного додавання товару.

**Корінь проблеми знайдено в `PromFields.tsx`:** при виборі групи з живого пошуку
(`/api/prom/groups`) в `group_name` записувався `g.fullPath` — повний шлях з ієрархією
через стрілку, наприклад "Голови жоржин, хризантем → Голови георгін, хризантем". Цей
текст потім писався як буквальна назва в `<category id="X">...</category>` в YML.
Оскільки такого тексту (зі стрілкою) не існує серед реальних назв груп в акаунті Prom,
Prom не міг розпізнати це як вже існуючу групу за назвою — і при кожному новому імпорті
файлу створював нову групу-дублікат замість перевикористання існуючої.

**Виправлення:**
- `group_name` тепер зберігає ВЛАСНУ назву групи (`g.name`), без шляху. `fullPath`
  лишився тільки для відображення в UI (пошук/вибрана позначка), де ієрархія потрібна
  для орієнтації користувача.
- Додав нове поле `group_parent_id` в `PromData` — id батьківської групи.
- В `lib/generators/prom-yml.ts` доданий атрибут `parentId` до тега `<category>`:
  `<category id="X" parentId="Y">Власна назва</category>` — саме так, за офіційною
  документацією Prom, і має передаватись вкладеність (а не через конкатенацію назв).

**Перевірено живим тестом:** обрав групу "Голови георгин, хризантем" (вкладена в "Голови
жоржин, хризантем") — згенерований YML тепер містить
`<category id="126856271" parentId="31749455">Голови георгин, хризантем</category>` —
правильний id, правильний parentId, і назва без стрілки. Прибрав тестовий пост.

⚠️ Важливо: групи-дублікати, які вже встигли створитись в акаунті Prom ДО цього фіксу,
самі не зникнуть — це виправляє тільки причину появи НОВИХ дублікатів. Наявні дублікати
доведеться прибрати вручну в кабінеті Prom (Товари → Групи), об'єднавши/видаливши зайві.

## Сесія 2026-07-05 (продовження 5) — реальний тест показав: моя порада про мову була невірна

Юзер зробила реальний імпорт в Prom і показала скріншоти кабінету + сам YML-файл.
Виявилось:

1. **Назва/Опис/Пошукові запити потрапляли в РОСІЙСЬКЕ поле, а УКРАЇНСЬКЕ лишалось
   порожнім.** Це прямо суперечить тому, що я написав в попередній сесії (нібито Prom

   **Виправлення:** повернув дублювання — `<name_ua>`, `<description_ua>`,
   `<keywords_ua>` тепер знову заповнюються тим самим текстом, що і `<name>`,
   `<description>`, `<keywords>` (`lib/generators/prom-yml.ts`). Мінус: в кабінеті
   Prom "російська" версія буде показувати помітку "містить слова українською" з
   кнопкою "Перекласти" — це РУЧНА кнопка на кожен товар, автоматичного перекладу
   під час імпорту файлу немає. Але головне — українська версія (яку бачать
   відвідувачі за замовчуванням) тепер точно заповнена.

2. **Категорія на маркетплейсі не підтягнулась** (Prom підставив хибну "М'які
   іграшки" замість очікуваної категорії штучних квітів), хоча юзер вказала
   `category_url`. З YML-файлу видно причину-кандидата: посилання було скопійоване
   з мовним префіксом `/ua/` (`prom.ua/ua/Iskusstvennye-tsvety-...`) — можливо, Prom
   матчить категорію за канонічним посиланням БЕЗ цього префіксу. Додав автоматичне
   прибирання `/ua/` з URL перед відправкою (`lib/generators/prom-yml.ts`) — це
   гіпотеза, не підтверджена офіційно, тому потрібна перевірка юзером на наступному
   реальному імпорті. Якщо не допоможе — треба буде спробувати `portal_category_id`
   (числовий) замість URL, або змиритись з автовизначенням Prom і вручну
   виправляти хибні категорії (Prom підтримує скаргу на категорію в кабінеті).

**Перевірено живим тестом (тільки структуру YML, не реальний імпорт в Prom):**
згенерований файл тепер містить `name_ua`/`description_ua`/`keywords_ua` з тим самим
текстом, і `portal_category_url` без `/ua/` префіксу. Прибрав тестовий пост.

## Сесія 2026-07-05 (продовження 6) — групи Prom: відмова від YML-підходу; Horoshop CSV пояснення

Юзер зробила ще один реальний тест: підтвердила, що з'явився **новий** (третій) дублікат
групи в Prom, попри фікс з `parentId` + власною назвою групи з минулої сесії. Також дала
скріншоти реального імпорту CSV в Horoshop.

### 1. Групи на Prom — прибрав спробу автоматичної прив'язки через YML

Два попередні фікси (спочатку прибрати "шлях зі стрілкою" з назви, потім додати
`parentId`) не допомогли — юзер підтвердила, що з'явився новий дублікат навіть з
правильними `id`+`parentId`+назвою. Висновок: ручний "завантажити файл" імпорт в кабінеті
Prom, вочевидь, не запам'ятовує зв'язок `id ↔ вже існуюча група` між окремими сесіями
імпорту (на відміну від сталого підключеного фіда) — тобто надійного способу зробити
це через одноразові YML-файли, схоже, немає.

Щоб не продовжувати плодити дублікати на живому акаунті, **прибрав відправку групи в
Prom повністю**: `<categories>`/`<categoryId>` для групи більше не генерується
(`categoryId` завжди `"1"`, як і до появи цієї фічі). Поле "Група на сайті" в формі
лишив — але тепер це просто підказка-нагадування собі, яка позначена явним
попередженням в UI ("не передається в Prom") — призначення товару в групу треба робити
вручну в кабінеті Prom.

Також попутно виправив: `offerId` (id офера, за яким Prom розпізнає товар для
оновлення/дубля) раніше генерувався як `post-${Date.now()}` коли SKU не вказано — тобто
ЩОРАЗУ при повторному скачуванні того самого поста виходив РІЗНИЙ id, і Prom, найімовірніше,
створював ще один окремий товар замість оновлення існуючого (ще одне ймовірне джерело
"дублювання"). Тепер id стабільний: `PB-{перші 8 символів id поста}` (той самий пост
завжди дає той самий offer id). Перевірено: два скачування підряд дають однаковий id.

⚠️ Це не остаточне рішення проблеми груп — це відмова від ненадійного підходу, щоб
зупинити подальше засмічення акаунту. Існуючі дублікати (тепер їх 3) треба прибрати
вручну в кабінеті Prom (Товари → Групи).

### 2. Horoshop CSV — пояснення (не баг коду)

Дослідив офіційну документацію Horoshop (help.horoshop.ua) щодо реальних скріншотів
юзер:

- **Фото ніколи не імпортуються автоматично** — це офіційно задокументована поведінка
  Horoshop для БУДЬ-ЯКОГО CSV/Excel імпорту, не залежить від нашого файлу: "Attention!
  Columns containing links to photos (Image, Gallery, 360-degree view) are never
  imported by default. You must match them manually." Юзер щоразу має вручну вибрати
  "Image"/"Gallery" для колонки "Фото" в прев'ю імпорту, і обрати "Rewrite" чи "Add" в
  діалозі дій. Наш формат колонки (кілька посилань через `; `) — правильний, офіційно
  задокументований спосіб для кількох фото в одній комірці.
- **Категорія — обов'язкове поле** для НОВОГО товару (разом з Артикулом і Назвою) —
  раніше в нашій формі "Розділ каталогу" був позначений як необов'язковий, хоча Horoshop
  вимагає його для успішного імпорту нового товару. Виправлено: змінив підпис поля в
  `HoroshopFields.tsx` (тільки для Basic/Standard/CSV-варіанту, не для Pro/API) на
  явне попередження, що це обов'язково для CSV-імпорту. Додав попередження у валідацію
  форми (`PostForm.tsx` → `getValidation`): якщо обрано Horoshop і розділ не вказано —
  тепер показується "не вказано розділ каталогу" в блоці перевірки платформ.
- **"Не все співпадає автоматично, поля російською"** — Horoshop автоматично мапить
  колонку CSV на поле товару, ТІЛЬКИ якщо назва колонки збігається з назвою поля в
  **шаблоні товару категорії** (Довідники → Шаблони товарів), який юзер сама налаштовує
  в своєму кабінеті Horoshop — це НЕ фіксований загальний словник Horoshop і не залежить
  від мови нашого файлу. Судячи зі скріншоту, її шаблон містить поля з російськими
  назвами ("Родительский артикул", "Название модификации (UA)" тощо) — тому наші
  українські заголовки не мапляться автоматично для полів, яких немає точного
  збігу. Це налаштування на боці її акаунту Horoshop, ми не можемо це "полагодити" з
  коду. Хороша новина: Артикул/Валюта/Кількість вже мапляться автоматично в її
  прев'ю — це означає, що Horoshop ЗАПАМ'ЯТОВУЄ зіставлення колонок з попередніх
  імпортів; отже інші поля (Назва, Ціна, Наявність, Стан, Колір, Матеріал, Опис, Фото,
  SEO ключові слова) потрібно буде зіставити вручну лише ОДИН РАЗ (з поточними,
  стабільними заголовками нашого файлу) — і надалі це також має запам'ятатись.

Джерела: help.horoshop.ua/en/articles/1684865 (Importing products from a file),
help.horoshop.ua/en/articles/1703614 (Preparing files for importing products).
